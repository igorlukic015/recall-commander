# Recall Commander - Artifact Lifecycle

## Core Principle

Generated artifacts are Markdown-first learning documents, not database exports.

They are:

- human readable
- editable with any Markdown editor
- portable
- understandable years later
- independent from Recall Commander internals

The database stores metadata only.

The Markdown files are the learning history.

## The Lifecycle

The full workflow, as implemented:

```
Question Block          (user-authored, inside any Markdown note)
      |
      |  rc scan
      v
Question                (in-memory only; re-parsed on every scan)
      |
      |  rc assessment create
      v
Assessment              (RC-generated snapshot: Assessments/assessment-YYYY-MM-DD-NNN.md)
      |
      |  user: Save As + fill in answers
      v
Attempt                 (user-authored copy with answers; RC never writes it)
      |
      |  rc review create
      v
Review                  (RC evaluation artifact: Reviews/review-YYYY-MM-DD-NNN.md)
```

Each stage creates a more complete historical artifact. Every generated
artifact is self-contained: content is always copied forward, never
referenced, so deleting or editing an earlier stage never invalidates a
later one.

## Stage 1: Question Block

A Question Block is a `:::rc-question` container embedded in any Markdown
document inside a registered Question Source. The syntax is specified in
[Question Block.md](Question%20Block.md).

Question Blocks are the only part of the user's files Recall Commander
reads. They are the user's property and are never modified.

## Stage 2: Question

`rc scan` recursively parses every Markdown file in every registered source
and turns valid Question Blocks into in-memory `Question` objects (type,
prompt, optional reference answer, concepts).

Questions are deliberately transient:

- they are **never persisted** — not in SQLite, not in any cache
- every scan re-parses the sources from scratch
- there are no question IDs (see "Question Identity Decision" below)

Invalid blocks are skipped and reported as warnings with file and line
number; a malformed question never stops a scan.

## Stage 3: Assessment

### Purpose

An Assessment is a generated snapshot of questions selected at a specific
moment.

It answers:

> "What questions did Recall Commander ask me?"

### How it is created

`rc assessment create [--count N]` (default 10 questions):

1. Scans all sources (a fresh scan every time).
2. Selects questions at random, without duplicates
   (`RandomQuestionSelector` behind `IQuestionSelector`).
3. Snapshots only the prompts into an `Assessment`.
4. Writes `Assessments/assessment-YYYY-MM-DD-NNN.md` under the current
   working directory.

### Important Properties

An Assessment:

- is independent from Question Sources
- contains copied question text
- does not depend on the original question files
- carries its own artifact id in frontmatter
- remains readable years later

If a user deletes or modifies the original question source, old assessments
remain valid.

### Assessment File Format

Exactly what the assessment renderer produces:

```markdown
---
type: assessment
id: assessment-2026-07-19-001
title: Assessment 2026-07-19
created: 2026-07-19T17:05:09
question_count: 2
---

# Assessment 2026-07-19

This assessment was generated by Recall Commander.

---

## Question 1

Explain how garbage collection works in .NET.

Your answer should include generations and why they exist.

### Answer

---

## Question 2

What is boxing in C#?

### Answer
```

Frontmatter keys are snake_case; timestamps use `yyyy-MM-ddTHH:mm:ss` (UTC).
The `id` equals the file name without extension — the artifact carries its
identity inside the document, so later artifacts can reference it without
depending on file paths or database records.

Every question ends with an empty `### Answer` heading, ready for the user
to fill in.

### What an Assessment Contains

Contains:

- frontmatter (`type`, `id`, `title`, `created`, `question_count`)
- a title heading and a short instruction line
- the question prompts
- an empty `### Answer` heading per question

Does not contain:

- reference answers
- question types
- concepts
- source paths
- review information

Question metadata is deliberately invisible: the user only sees the prompt,
never `Type: Explanation` or a concept list.

## Stage 4: Attempt

### Purpose

An Attempt represents the user's completed answers.

It answers:

> "What did I answer when I took this assessment?"

### User Workflow

Recall Commander is not involved while the user answers:

1. Generate an assessment, e.g. `Assessments/assessment-2026-07-19-001.md`.
2. **Save As** a copy, e.g. `assessment-2026-07-19-001.attempt.md`
   (any name and location work — RC finds attempts because the user hands
   them to a command, not by convention).
3. Fill in answers under each `### Answer` heading, in any Markdown editor.
4. Hand the file to `rc attempt validate` (optional check) or
   `rc review create`.

### Important Properties

An Attempt:

- is authored by the user, never generated or modified by RC
- contains the original questions and the user's answers
- keeps `type: assessment` in its frontmatter — a Save As does not change
  the artifact type; a file is an attempt because the user hands it to the
  attempt parser
- preserves the assessment's `id`, which becomes the attempt's link back to
  the assessment
- remains a historical record; the original assessment stays untouched

An empty answer is valid: it means the question was left unanswered.

### Attempt Format

An attempt is an assessment file with answers written in:

```markdown
---
type: assessment
id: assessment-2026-07-19-001
title: Assessment 2026-07-19
created: 2026-07-19T17:05:09
question_count: 2
---

# Assessment 2026-07-19

This assessment was generated by Recall Commander.

---

## Question 1

Explain how garbage collection works in .NET.

Your answer should include generations and why they exist.

### Answer

The GC frees unreachable objects. Generations exist so short-lived objects
are collected cheaply.

---

## Question 2

What is boxing in C#?

### Answer

Boxing converts a value type into an object on the heap.
```

### What the Attempt Parser Requires

The attempt parser (`rc attempt validate`, and the first step of
`rc review create`) accepts the structure the assessment renderer produces:

- YAML frontmatter with `type: assessment` and a `created` timestamp in the
  format `yyyy-MM-ddTHH:mm:ss`
- a title: a `title` frontmatter field or the first `# ` heading
- `## Question ...` headings starting each question section
- a `### Answer` heading inside every question section; everything after it
  (until the next question) is the answer
- the assessment identity is read from an `assessment` or `id` frontmatter
  field; both are optional

Top-level `---` separators between questions are ignored, so a `---` inside
an answer survives. A second `### Answer` inside an answer is treated as the
user's own Markdown. Any other `#`/`##` heading ends the current question
section.

All problems are collected and reported together with line numbers; a
partially parsed attempt is never accepted, because it would silently
misrepresent what the user answered.

### Answer Separator Decision

Use `### Answer`, not `:::rc-answer`.

The Attempt is a human document, not a Question Source. It does not need RC
syntax.

## Stage 5: Review

### Purpose

A Review is the evaluation of an Attempt.

It answers:

> "How well did I understand the material?"

### How it is created

`rc review create <attempt-file>`:

1. Reads and parses the attempt (same parser as `rc attempt validate`;
   invalid attempts are rejected with diagnostics).
2. Evaluates every question/answer pair through `IQuestionEvaluator` — the
   deterministic fake by default, or an AI provider when configured (see
   [AIArchitecture.md](AIArchitecture.md)).
3. Assembles a `Review` snapshot and writes
   `Reviews/review-YYYY-MM-DD-NNN.md`.

The attempt file stays untouched.

### Important Properties

A Review:

- contains every question and answer copied from the attempt
- adds an evaluation after each answer
- records which evaluator produced it (`evaluator` frontmatter,
  e.g. `fake` or `ollama/llama3.2`)
- references the attempt only by artifact id (`attempt` frontmatter);
  the content itself is copied, so the review stays readable if the attempt
  disappears
- is a learning artifact itself — the full context matters, not just a
  summary

### Review File Format

Exactly what the review renderer produces:

```markdown
---
type: review
id: review-2026-07-19-001
attempt: assessment-2026-07-19-001
title: Review - Assessment 2026-07-19
created: 2026-07-19T17:05:23
evaluator: fake
question_count: 2
---

# Review - Assessment 2026-07-19

## Overall Summary

2 of 2 questions were answered.

---

# Question 1

## Question

Explain how garbage collection works in .NET.

Your answer should include generations and why they exist.

## Answer

The GC frees unreachable objects. Generations exist so short-lived objects
are collected cheaply.

## Evaluation

Score: 8/10

Understanding: Strong

The answer demonstrates a good understanding.

### Strengths

- The core idea is explained clearly.

### Missing Information

- A concrete example.

### Incorrect Statements

None.

### Suggestions

- Add an example to strengthen the answer.
```

Per question, the evaluation carries:

- **Score** — integer 0 to 10
- **Understanding** — Poor, Weak, Partial, Good, Strong or Excellent
- a one-or-two sentence summary
- **Strengths**, **Missing Information**, **Incorrect Statements**,
  **Suggestions** — bullet lists; an empty list renders as `None.` so the
  document reads completely on its own

An unanswered question renders `*No answer was provided.*` in place of the
answer and is scored 0 / Poor.

## Artifact Naming and Storage

All generated artifacts share one writing pipeline
(`ArtifactWriter<T>` → `IArtifactRenderer<T>` → `IArtifactStore`):

- File names are `{slug}-{yyyy-MM-dd}-{NNN}.md` — slug (`assessment`,
  `review`), UTC creation date, and the first free three-digit sequence
  number counting up from `001`.
- The artifact id is the file name without extension and is embedded in the
  frontmatter; the store assigns the name and passes it to the renderer, so
  the two can never disagree.
- Assessments go to `Assessments/`, reviews to `Reviews/`, both under the
  directory the command was invoked from.
- Existing files are **never overwritten** — artifacts are historical
  records.

## Database Responsibility

SQLite does not store the artifacts.

It currently stores:

- registered question sources (`question_sources`)
- application settings (`app_settings`, currently the schema version)

It stores no question text, no answers, no review content. Even artifact
metadata tables are not needed yet: the artifact files on disk are the
record, and tools like the Workbench list them straight from the filesystem.

## Question Identity Decision

Important decision:

No question IDs in MVP.

Reasons:

- questions live in user-controlled files
- files move
- lines change
- content changes
- sources appear/disappear

Trying to permanently track questions creates complexity.

### MVP Approach

Every scan:

```
Scan sources
        |
        v
Parse Question Blocks
        |
        v
Create current question collection
```

No synchronization. No question history. No identity.

Artifacts, in contrast, do have identity (`assessment-2026-07-19-001`) —
they are generated, immutable and sequenced, so an id is stable and cheap.
The id lives in the document itself, not in a database.

## Source Philosophy

Question Sources are just inputs.

Examples:

- `~/ObsidianVault`
- `~/ProgrammingQuestions`
- `~/Research`

RC does not care about folder organization. A valid source could contain
`everything.md` with biology, programming and history questions mixed
together. The user decides organization.

Overlapping sources are handled: a file reachable from two registered
sources is scanned once.

## Future Considerations

These decisions leave room for:

- concept graphs
- scheduling
- adaptive testing
- richer LLM evaluation (reference answers, concepts, history)
- statistics
- community sharing

without forcing MVP complexity.

## Final Artifact Model

```
Question Sources
(User owned)
        |
        v
Question Blocks
(User authored)
        |
        v
Assessment
(RC generated snapshot)
        |
        v
Attempt
(User completed assessment)
        |
        v
Review
(RC evaluation artifact)
```

The core philosophy:

> Knowledge belongs to the user. Assessment belongs to Recall Commander. Learning history belongs to Markdown artifacts.
